Design Philosophy

"Good AI system design is knowing when not to use AI at all."

The most consequential decision in any AI workflow is not which model to use — it is how sparingly to use AI at all. Every inference call carries latency, cost, and failure risk. Rule-based logic is fast, free, and deterministic.

A well-designed system reserves AI for genuine ambiguity: multi-turn conversations, context-dependent responses, and judgment calls. Everything predictable gets a rule. The result scales cleanly, fails predictably, and is easy to audit.

01

Separate decision logic from generation

Rule-based systems handle predictable decisions. AI is reserved for tasks that require interpretation, context, or generation.

02

Use AI only where variability exists

Simple interactions are handled with fixed responses. AI activates only when the input is genuinely unpredictable.

03

Define clear completion states

Systems do not rely on "good enough." A process is complete only when it meets explicit, structural criteria.

04

Design for failure, not just success

Identify where failure can occur, then isolate those points to prevent full workflow breakdown.

05

Control cost through architecture

A lightweight classifier handles the majority of interactions. Heavy agents only trigger when necessary.

Instagram DM & Comment
Automation , Scroll & Tell

Multi-turn conversation pipeline with structured story extraction. From first comment to complete story, fully automated.

+

The Problem

Scroll & Tell's content is the community's stories. Collecting them manually is a bottleneck: reply to a comment, start a DM, ask the right questions, keep the conversation going, then save the result. Most tools like ManyChat handle this with keyword triggers and fixed responses — which works for giveaways, but not for extracting a detailed personal story through a real conversation.

↓

↓

↓

The Core Innovation

The system shifts from trigger-based automation to goal-oriented conversation design. Most DM tools respond to messages with fixed outputs. This system treats each conversation as a structured process with a defined objective: extract a complete, coherent story.

A lightweight TinyLlama classifier (temperature 0, 512 context) handles 80 to 90% of conversations at near-zero cost. Only messages requiring context, follow-up, and judgment reach the Phi-3 Mini agent. That separation is a deliberate cost and performance decision, not a default.

Honest Limitations

Classification reliability: TinyLlama degrades on slang, mixed-intent messages, and non-English text. No confidence scoring or misclassification logging exists.

Google Sheets bug: The Save Story node has empty column mapping. It executes without error but writes no data.

In-memory conversation state: If n8n restarts, all active threads lose context. Production requires Redis or database-backed memory.

No webhook signature validation: The endpoint does not verify Meta's X-Hub-Signature-256. Any actor who discovers the URL can inject payloads.

No deduplication: Instagram can fire duplicate webhook events. One user message could trigger multiple replies.

Incomplete error coverage: API error checking covers only the canned DM path. Other paths fail silently.

How It Evolves to Production

01

Replace TinyLlama with regex and keyword matching for simple intents, reserve LLM for genuinely ambiguous messages.

02

Add error recovery: wrap Ollama calls in try/catch with a graceful fallback message.

03

Move story storage to Supabase or Airtable with deduplication, tagging, and a review interface.

04

Add deduplication: hash senderId and messageText and timestamp, skip if seen in the last 60 seconds.

05

Persist conversation context in a database keyed to senderId.

06

Build a story review queue: dashboard showing completed stories, flagging incomplete ones for human review.

4-Module AI Content
Pipeline

End-to-end automation from Instagram DM intake to multi-platform publishing using self-hosted LLMs and modular n8n workflows.

+

The Problem

Scroll and Tell is a Reddit-style storytelling brand on Instagram. The manual workflow for sourcing, processing, approving, and publishing each story involved too many disconnected steps: monitor comments, start a DM conversation, ask the right questions, extract the story, write content, get it approved, then publish across platforms.

Each step is repeatable with clear inputs and outputs. That means each step can be automated — if the system is designed correctly.

System Overview

Module 1 — DM & Comment Intake

Instagram webhook triggers intent classification (TinyLlama). Story submissions activate the AI Story Collector Agent (Phi-3 Mini). Extracts story with structured conversation, saves to Google Sheets.

↓

Module 2 — Content Engine

Webhook trigger on story completion. Duplicate check against PostgreSQL, then Phi-3 generates JSON: hook, scene breakdown, caption variants, hashtag clusters. Insert to DB as pending_approval.

↓

Module 3 — Approval System

Telegram message with Approve/Reject/Upload buttons. Human reviews with one tap. Database status updates to approved or rejected.

↓

Module 4 — Distribution Engine

Retrieves approved content from DB, fans out to Instagram and Facebook via Graph API. Rate limits and error handling built in.

Module 1 — DM & Comment Intake

Instagram sends all events (DMs and comments) to a single webhook. First decision: is this a DM or a comment? — field check, no AI needed. Then TinyLlama (temp 0) classifies intent into six categories.

Five intents get rule-based canned replies. One intent — story_submission — activates Phi-3 Mini (via Ollama) for a multi-turn conversation. Completion detected via string match on STORY_COMPLETE marker.

Webhook verification Rule Binary check, deterministic
DM vs Comment routing Rule Structural field, no ambiguity
Intent classification AI Natural language varies
Canned replies Rule Fixed brand voice, no inference cost
Story conversation AI Requires context and judgment
Completion detection Rule String match on structured marker

Module 2 — Content Engine

Duplicate check against PostgreSQL using a hash of story fields — stops immediately if already processed, no wasted inference. Phi-3 generates structured JSON output (hook, scene breakdown, caption variants, hashtags).

Output is parsed and validated. Failed parses are logged to PostgreSQL with notification — no silent failures. On success, inserts to DB as pending_approval and triggers Module 3.

Module 3 — Approval System

Telegram message with hook, caption, and three inline buttons: Approve, Reject, Upload. Callback routes to three paths: updates DB to approved/rejected, or saves media path for manual upload.

Human approval is structurally required — Module 4 cannot fire without approved status. One tap from anywhere.

Module 4 — Distribution Engine

Retrieves approved content package from DB, fans out to platform branches. Instagram creates media container via Graph API, waits 30s for processing, then publishes. Same for Facebook. TikTok and YouTube branches exist but deactivated.

Error trigger catches any platform API failure, logs to PostgreSQL, fires notification. Every status transition is persisted for auditability.

Facebook publish Rule Isolated branch, failure contained
Content generation AI Structured output needs LLM
Approval routing Rule Switch node, deterministic

What This Is Not

This is not a social media scheduling tool. It is a demonstration of multi-module AI workflow architecture under real constraints: self-hosted infrastructure (no per-token cost), modular failure isolation (4 workflows = 4 failure surfaces), selective AI use (only 2 calls in entire pipeline), and human-in-the-loop by design (structural, not bolted on).

Key Takeaway

Four decoupled modules means each can be tested independently, replaced without touching others, and scaled without risking the whole pipeline. The self-hosted LLM constraint forced tighter prompt engineering — Phi-3 Mini is not GPT-4, so prompts had to be specific and output formats enforced. That constraint made the system better.

Live Prompt Engineering Content System Design

AI-Assisted Short-Form
Content Automation Workflow

A modular, 5-stage prompt pipeline that transforms raw Reddit stories into production-ready video content packages, reducing prep time from 45 to 15 minutes.

+

The Problem

Reddit storytelling content works on short-form video when it preserves emotional pacing, character clarity, and platform-specific voice. Most creators either post the raw thread unedited or pass it through a single AI prompt — both produce inconsistent results. A single pass cannot simultaneously optimize for caption length, hook strength, CTA relevance, and hashtag fit.

Why Modular Over Single-Prompt?

Each stage enforces a different constraint: chronological logic, character-count awareness, taxonomy thinking. Combining these in one prompt creates conflicts — the model optimizes for too many things at once and drops constraints at the edges.

The 5-Stage Pipeline

Stage 1 , Story Extraction & Restructuring

Strips filler, reorders events chronologically, preserves emotional arc. Output: beat-based narration script and editor notes.

↓

Stage 2 , Script Adaptation

Converts restructured story into short-form script timed for TikTok/Reels/Shorts pacing (60 to 90 seconds).

↓

Stage 3 , Caption Generation

Produces hook, body, and CTA captions within platform character limits. Maintains brand voice, avoids spoilers.

↓

Stage 4 , CTA Selection

Chooses the appropriate call-to-action based on story type (relate, share, react, follow). One CTA only.

↓

Stage 5 , Hashtag Classification

Outputs curated hashtag set organized by discovery tier (broad, niche, branded). Topic-aligned, not trend-chasing.

Honest Limitations

This is a human-in-the-loop workflow, not an autonomous pipeline — a human reviews each stage before the next runs. It requires reasonably coherent input; heavily fragmented posts need manual cleanup before Stage 1 produces usable output. There is no feedback loop yet between content performance and prompt refinement.

Impact

65%

Reduction in per-video content prep time

5

Isolated stages, errors stay contained

3

Platforms: Instagram, TikTok, YouTube Shorts

↑

Output consistency across batches significantly improved

Design Principles Applied

→

No stage does more than one thing, each has a defined input, a constrained output, and a specific job.

→

Guardrails maintain story-specific tone, the hook reflects what actually makes the story shareable, not a template phrase.

→

The output of each stage feeds the next, so errors stay contained and adjustments are isolated.

→

Final assembly produces a production-ready package for direct handoff into video editing and publishing.

Key Takeaway

This is not a content brand case study. It is a workflow design case study demonstrating how modular AI systems can transform unstructured inputs into consistent, production-ready outputs under real platform constraints, with a human in the loop to catch what the system cannot.

MathDesk — AI-Assisted
Math Learning Platform

A solo-built, free math web app with an AI tutor, camera scan, handwriting input, and interactive learning tools. Self-hosted n8n backend. No paywall, no account required.

+

Overview

MathDesk is a free AI-assisted math learning web app designed to make problem-solving and concept learning more accessible. It combines a multi-mode AI tutor, multimodal input methods (camera, handwriting canvas, image upload), interactive math tools, and a curated resource directory into a single browser-based experience.

Built as both a technical exploration and a product-focused systems project. The design constraint was simple: no paywall, no ads, no mandatory sign-up. Every feature had to be accessible without payment or account creation.

The Problem It Addresses

Many existing math tools lock step-by-step explanations behind subscriptions, require account creation before showing anything useful, or focus on final answers without addressing the reasoning. MathDesk was built to explore what a more accessible approach looks like: AI tutoring, visual input methods, and learning tools in one lightweight, free platform.

System Architecture

Frontend

Single index.html — vanilla HTML, CSS, and JavaScript. No framework, no build step. Camera APIs, canvas drawing, drag-and-drop, clipboard paste, modal systems, and draggable UI components implemented without React or any abstraction layer.

↓

Automation Layer (n8n, self-hosted in Docker)

Multi-branch webhook workflow routes by mode (Solve, Learn, Practice, Deskbot), builds the Groq API request, parses the response, and returns plain text to the frontend. Exposed via ngrok tunnel.

↓

LLM Layer (Groq API)

Text model: llama-3.3-70b-versatile. Vision model: llama-4-scout-17b — selected automatically when image input is detected (camera, handwriting canvas, upload).

↓

Auth & Storage (Supabase)

Optional user accounts with Row Level Security. Chat history saved per-user. The app works fully without login — auth is additive, not a gate.

AI Mode Design

Solve Mode AI Structured output: answer + steps table + hint + 2 practice problems
Learn Mode AI Conversational tutor voice, concept-first, examples woven in
Practice Mode AI Generates 5 problems by difficulty, checks submitted answers
Deskbot (App FAQ) AI Separate system prompt scoped to app usage only — cannot solve math
Model selection Rule Switches to vision model automatically when image payload is present
Webhook routing Rule n8n Switch node branches on the mode field — no ambiguity

My Role

I independently designed the system architecture, built the frontend, configured and managed the n8n AI workflows, integrated the Groq APIs, implemented Supabase auth and database storage, handled deployment and DNS configuration, and iterated through infrastructure debugging challenges throughout. AI-assisted development tools were used to accelerate UI iteration and workflow refinement. All integration, deployment, troubleshooting, and system decisions were handled independently.

Honest Limitations

Webhook instability: The n8n instance runs locally in Docker. When Docker restarts, the ngrok URL changes and the app breaks until manually updated. A Cloudflare Tunnel would fix this — planned but not yet done.

No endpoint authentication: The webhook is public. Any actor with the URL can send requests. There is no auth layer on the n8n webhook itself.

Groq free-tier ceiling: 14,400 requests/day. Under sustained load it would hit limits, with no graceful degradation currently built in.

No feedback loop: No mechanism to capture which responses were helpful. Prompt refinement is currently manual and intuition-driven.
