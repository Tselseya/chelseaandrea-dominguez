Design Philosophy

"Use AI where intelligence is genuinely required. Use rules everywhere else."

The most consequential decision in any AI workflow isn't which model to use — it's how sparingly to use AI at all. Every AI call carries latency, cost, and failure risk. Rule-based logic is fast, free, and deterministic.

A well-designed system reserves AI for genuine ambiguity: multi-turn conversations, context-dependent responses, judgment calls. Everything predictable gets a rule, a static response, or a binary check.

The result is a workflow that scales cleanly, fails predictably, and is easy to audit — because most of it isn't magic.

01

Separate decision logic from generation

Rule-based systems handle predictable decisions. AI is reserved for tasks that require interpretation, context, or generation.

02

Use AI only where variability exists

Simple interactions are handled with fixed responses. AI activates only when the input is genuinely unpredictable.

03

Define clear completion states

Systems don't rely on "good enough." A conversation is complete only when it meets explicit structural criteria.

04

Design for failure, not just success

Identify where failure can occur, then isolate those points to prevent full workflow breakdown.

05

Control cost through architecture

A lightweight classifier handles the majority of interactions. Heavy AI agents only trigger when necessary.

Live in Production n8n · Meta Graph API · Ollama AI Conversation Design

Instagram DM & Comment
Automation — Scroll & Tell

Multi-turn conversation pipeline + structured story extraction system. From first comment to complete story, fully automated.

+

The Problem

Scroll & Tell runs on audience participation — the content is the community's stories. But collecting those stories manually at scale is impossible: reply to a comment, start a DM, ask the right questions in the right order, keep the conversation going until the story is complete, then save it somewhere useful.

Most brands solve this with tools like ManyChat — keyword triggers that send fixed responses. That works for giveaways and link delivery. It doesn't work when your goal is to extract a detailed personal story through a multi-turn conversation.

System Architecture

Input Layer

Instagram Webhook — DMs + Comments via Meta Graph API

↓

Decision Layer

Event routing (DM vs Comment) → Intent classification (TinyLlama, temp 0) → 6 rule-based response paths

↓

Intelligence Layer

AI Story Collector Agent (Phi-3 Mini) → Per-user conversation memory → Story completion detection

↓

Output Layer

DM responses + Public comment replies + Structured story storage (Google Sheets)

Rule-Based vs. AI-Driven Logic

Webhook verification Rule Binary check, no ambiguity
DM vs Comment routing Rule Structural field check
Intent classification AI Natural language varies too much
Canned replies (5 intents) Rule Fixed, polished brand voice
Story extraction conversation AI Requires context, follow-up, judgment
Story completion detection Rule String match on structured marker

AI Agent — Story Complete

The Core Innovation

The system shifts from trigger-based automation to goal-oriented conversation design. Most DM tools respond to messages with fixed outputs. This system treats each conversation as a structured process with a defined objective: extract a complete, coherent story.

A lightweight TinyLlama classifier (temperature 0, 512 context) handles 80–90% of conversations at near-zero cost. Only messages requiring context, follow-up, and judgment reach the Phi-3 Mini agent. That separation is a deliberate cost and performance decision — not a default.

Honest Limitations

Classification reliability: TinyLlama degrades on slang, mixed-intent messages, and non-English text. No confidence scoring or misclassification logging exists. Unrecognized intents fall through to the AI agent, which may be inappropriate behavior.

Google Sheets bug: The Save Story node has empty column mapping — it executes without error but writes no data. Story capture requires proper schema mapping to function.

In-memory conversation state: If n8n restarts, all active threads lose context. Production requires Redis or database-backed memory.

No webhook signature validation: The endpoint doesn't verify Meta's X-Hub-Signature-256. Any actor who discovers the URL can inject payloads.

No deduplication: Instagram can fire duplicate webhook events. One user message could trigger multiple replies.

Incomplete error coverage: API error checking covers only the canned DM path. The AI story reply, public comment reply, and commenter DM fail silently.

How It Evolves to Production

01

Replace TinyLlama with regex + keyword matching for simple intents; reserve LLM for genuinely ambiguous messages only.

02

Add error recovery: wrap Ollama calls in try/catch with a graceful fallback message.

03

Move story storage to Supabase or Airtable with deduplication, tagging, and a review interface.

04

Add deduplication: hash senderId + messageText + timestamp, skip if seen in the last 60 seconds.

05

Persist conversation context in a database keyed to senderId rather than in-memory buffer.

06

Build a story review queue: dashboard showing completed stories, flagging incomplete ones for human review.

Positioning vs. ManyChat

ManyChat is excellent for what it does: delivering lead magnets, giveaways, broadcast sequences. This workflow does something different — it extracts content from users rather than delivering content to them. That requires a conversation that adapts, follows up, and knows when to stop. A keyword trigger can't do that. This is an AI-augmented alternative for a specific use case — multi-turn story collection — where rule-based automation reaches its limit.

Live Prompt Engineering Content System Design

AI-Assisted Short-Form
Content Automation Workflow

A modular, 5-stage prompt pipeline that transforms raw Reddit stories into production-ready video content packages — reducing prep time from 45 to 15 minutes.

+

The Problem

Reddit storytelling content works on short-form video platforms when it preserves emotional pacing, character clarity, and platform-specific voice. Most creators either post the raw thread unedited — losing momentum — or pass it to a single AI prompt and accept whatever comes out. Both produce inconsistent results.

The core issue: Reddit stories are unstructured. Variable length, scattered narrative order, buried context, off-topic tangents. A single AI pass can't simultaneously fix all of that while also optimizing for caption length, hook strength, CTA relevance, and hashtag fit.

Why Modular Over Single-Prompt?

Each stage enforces a different constraint. Story restructuring requires chronological logic. Caption writing requires character-count awareness. Hashtag classification requires taxonomy thinking. Combining these in one prompt creates conflicts — the model has to optimize for too many things simultaneously, and edge cases cause it to drop constraints.

The 5-Stage Pipeline

Stage 1 — Story Extraction & Restructuring

Strips filler, reorders events chronologically, preserves emotional arc. Output: beat-based narration script + editor notes.

↓

Stage 2 — Script Adaptation

Converts restructured story into short-form script timed for TikTok/Reels/Shorts pacing (60–90 seconds).

↓

Stage 3 — Caption Generation

Produces hook + body + CTA captions within platform character limits. Maintains brand voice, avoids spoilers.

↓

Stage 4 — CTA Selection

Chooses the appropriate call-to-action based on story type (relate / share / react / follow). One CTA only.

↓

Stage 5 — Hashtag Classification

Outputs curated hashtag set organized by discovery tier (broad, niche, branded). Topic-aligned, not trend-chasing.

Honest Limitations

This is a prompt-engineered system, not a fully automated one. A human reviews each stage's output before the next stage runs — making it a human-in-the-loop workflow, not an end-to-end autonomous pipeline.

It relies on the input story being at least reasonably coherent. Heavily fragmented posts or stories with missing context require manual cleanup before Stage 1 produces usable output.

There is no feedback loop yet between content performance (views, engagement, saves) and prompt refinement. That's the next evolution.

Isolated stages — errors stay contained

Design Principles Applied

→

No stage does more than one thing — each has a defined input, a constrained output, and a specific job.

→

Guardrails maintain story-specific tone — the hook reflects what actually makes the story shareable, not a template phrase.

→

The output of each stage feeds the next, so errors stay contained and adjustments are isolated.

→

Final assembly produces a production-ready package for direct handoff into video editing and publishing.

Key Takeaway

This is not a content brand case study. It's a workflow design case study demonstrating how modular AI systems can transform unstructured inputs into consistent, production-ready outputs under real platform constraints — with a human in the loop to catch what the system can't.
